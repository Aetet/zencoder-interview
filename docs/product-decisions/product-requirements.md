# Product Requirements: Agent Analytics Dashboard

**Product:** Organizational-level analytics dashboard for cloud-based AI coding agents
**Analogous to:** Analytics layer for a product like [Claude Code on the web](https://www.anthropic.com/news/claude-code-on-the-web)
**Date:** 2026-04-03
**Status:** Draft v2 (post-founder review)
**Input:** [Competitor Analysis](./competitor-analysis.md), [Task Brief](./task.md), [Reference Agent](./reference-agent/)

---

## 1. Problem Statement

Engineering organizations running AI coding agents in the cloud lack visibility into how those agents are used, what they cost, and whether they deliver value. Today, engineering leaders cannot answer basic questions:

- How much are we spending on AI agents this month?
- Which teams use agents effectively? Which don't?
- What's our ROI — are agents actually saving engineering time?

Without this visibility, organizations either overspend blindly or underinvest in a transformative capability.

## 3. Product Goals

1. **Cost transparency.** Make agent spending visible at org, team, and user levels. Competitors (Langsmith, Langfuse, Helicone) do basic cost tracking, but none attribute costs to teams with budget controls — this is our key differentiator.
2. **Adoption insight.** Show who uses agents, how, and how effectively — enabling data-driven rollout decisions.
3. **Actionable metrics.** Every metric on the dashboard should suggest a decision or action. No vanity numbers.

## 4. Agent Data Model (from Reference Agent)

The dashboard consumes telemetry from the reference agent runtime. All metrics must map to real data structures. Source: [`docs/reference-agent/`](./reference-agent/).

### 4.0.1 Token Usage (per assistant message)

```
TokenUsage {
  input_tokens: u64,                    // Tokens sent to model
  output_tokens: u64,                   // Tokens generated by model
  cache_creation_input_tokens: u64,     // Tokens written to cache (first use)
  cache_read_input_tokens: u64,         // Tokens read from cache (reuse)
}
```

**Cost implication:** Cache reads are ~10x cheaper than fresh input tokens. The ratio `cache_read / (cache_read + input)` is the **cache hit rate** — a key efficiency metric. High cache hit rate = the org is paying less per token of context.

### 4.0.2 Session Structure

A session = a complete conversation. Contains messages (system, user, assistant, tool). Usage is tracked **per assistant message**, enabling per-turn cost attribution.

Key fields for analytics:
- `messages[].role` — who sent the message
- `messages[].blocks[]` — content blocks (Text, ToolUse, ToolResult)
- `messages[].usage` — token counts (assistant messages only)
- `blocks[].tool_name`, `blocks[].is_error` — tool-level metrics

**Note:** The Session struct in the reference agent does not natively carry `team_id`, `user_id`, or `repository` fields. See Section 4.0.8 (Enrichment Pipeline) for how attribution metadata is attached.

### 4.0.3 Turn Summary (per turn)

```
TurnSummary {
  assistant_messages: Vec<ConversationMessage>,
  tool_results: Vec<ConversationMessage>,
  iterations: usize,          // API round-trips in this turn
  usage: TokenUsage,          // Cumulative for this turn
}
```

**`iterations`** = number of LLM API calls within a single turn. High iterations = agent is looping (tool call -> result -> tool call -> ...). This is the concrete measure of "agent efficiency per step."

### 4.0.4 Tool System (19 built-in tools)

| Permission Level | Tools |
|---|---|
| `ReadOnly` | `read_file`, `glob_search`, `grep_search`, `WebFetch`, `WebSearch`, `Skill`, `ToolSearch`, `Sleep`, `SendUserMessage`, `StructuredOutput` |
| `WorkspaceWrite` | `write_file`, `edit_file`, `TodoWrite`, `NotebookEdit`, `Config` |
| `DangerFullAccess` | `bash`, `PowerShell`, `REPL`, `Agent` |

Each tool call produces a `ToolResult` with `is_error: bool` and string output. Tool output size (string length) directly consumes context window.

### 4.0.5 Agent Status States

```
idle → thinking → using_tool → thinking → ... → completed
                                              → error
                                              → cancelled
              → waiting (for user permission)
```

7 states: `idle`, `thinking`, `using_tool`, `waiting`, `completed`, `error`, `cancelled`.

### 4.0.6 Error Taxonomy (30+ error codes)

| Category | Codes | Example |
|---|---|---|
| API/Auth | AUTH_001–004 | Missing credentials, expired token |
| Network | NET_001–004 | HTTP errors, retries exhausted, SSE frame errors |
| Runtime | RUNTIME_001–003 | Max iterations exceeded, empty response |
| Tool | TOOL_001–002 | Unknown tool, execution failed |
| Permission | PERM_001–004 | Mode denied, hook blocked, escalation denied |
| Session | SESSION_001–003 | IO error, corrupted file, format error |
| MCP | MCP_001–004 | Server not found, tool not found, connection failed |
| Plugin | PLUGIN_001–003 | Invalid manifest, load failed |
| Config | CONFIG_001–002 | Parse error, missing file |

Full error code reference: [`errors-in-example-agent.md`](./reference-agent/errors-in-example-agent.md)

### 4.0.7 Data Refresh Model

The agent runtime streams events via SSE (`TextDelta`, `ToolUse`, `Usage`, `MessageStop`) for its own conversation loop.

**V1 data refresh model:**
- **Costs, Teams, Settings** — standard request/response views. Data loads on page open and when the user changes filters. No polling, no streaming. The user sees a snapshot; to refresh, they change a filter or revisit the page.
- **Overview "Go Live" mode** — The Overview page has an opt-in "Go Live" button. When activated, the dashboard opens a dedicated SSE connection (`GET /api/overview/live`) that pushes aggregated KPI, chart, and leaderboard updates every 5 seconds. The connection closes when the user clicks "Stop Live", navigates away, or closes the tab. Only the Overview screen supports live mode.
- **Overview default** — When not in live mode, Overview behaves like other views: data loads on page open and filter change. No polling.

**V2:** Live session monitoring via direct SSE consumption for real-time agent-activity views.

### 4.0.8 Enrichment Pipeline (Prerequisite)

The raw Session struct does not include organizational metadata. Before session data reaches the dashboard, an enrichment step must attach:

| Field | Source | Notes |
|---|---|---|
| `team_id` | Identity provider / org management API | Maps the session-initiating user to their team |
| `user_id` | Authentication token on session creation | The user who started the session |
| `repository` | Working directory / git remote parsed from session context | Extracted from `bash` tool calls or workspace config |
| `model_id` | API request headers or response metadata | Which model was used (Haiku/Sonnet/Opus) |

**Implementation:** The ingestion service enriches each session at write time. This is prerequisite infrastructure work that must be completed before any team/user/repo filtering or attribution is functional. The enrichment pipeline is a hard dependency for FR-2 and FR-5.

## 5. Functional Requirements

### V1 Scope

V1 targets a single organization, up to 1,000 users, and 100,000 sessions/month. Ship in 6 weeks. The V1 feature set is: FR-1 (Org Overview), FR-2 (Cost & Usage), FR-5 (Team Breakdown), FR-7 (Filters), FR-11 (Budget Alerts), and a static Insights panel (replacing the full Optimization Advisor).

### 5.1 Dashboard Views

#### FR-1: Organization Overview (Landing Page)

The default view. Provides org-wide snapshot for leadership.

**Must show:**
- Total agent sessions this period (day/week/month toggle)
- Total cost this period (with trend vs. previous period)
- Active users count and adoption rate (users with agent activity / total org users)
- Session completion rate (sessions ending in `completed` status / total sessions)
- North star metric: cost per completed session, trending weekly

**Visualizations:**
- Activity trend chart (sessions over time, daily granularity)
- Cost trend chart (with budget line overlay if budget is set)
- Team leaderboard (ranked by configurable metric: cost, session count, completion rate)

#### FR-2: Cost & Usage Analytics

Dedicated cost view for budget owners and engineering leaders.

**Must show:**
- Total spend breakdown: by team, by user, by model
- Token consumption (4-dimensional, from `TokenUsage`):
  - `input_tokens` — fresh tokens sent to model (most expensive)
  - `output_tokens` — tokens generated by model
  - `cache_creation_input_tokens` — tokens written to prompt cache (one-time cost)
  - `cache_read_input_tokens` — tokens served from cache (~10x cheaper than input)
- **Cache efficiency panel:**
  - Cache hit rate: `cache_read / (cache_read + input_tokens)` — higher = cheaper
  - Cache savings: estimated cost if all cached reads had been fresh input tokens, minus actual cost
  - Cache hit rate by team — shows who benefits most from caching
  - Trend: cache hit rate over time (should increase as agents reuse context patterns)
- Cost per completed session
- Budget tracking: actual vs. budget, projected month-end spend
- Trend analysis: cost trajectory with 30-day projection

**Filtering:**
- By time range, team, user, model

**Deferred to V2:** Cost per repository, cost per PR, cost per lines changed (requires Git integration — see Section 7).

#### FR-3: Agent Activity & Throughput (V2)

Detailed view of what agents are doing, how fast, and how healthy they are. Deferred to V2 because the Org Overview and Cost views will cover 90% of V1 use cases.

**V2 scope includes:** Live agent status, health indicators, error breakdown panel, tool usage analytics, context usage analytics, session timeline traces.

#### FR-3.1: File Heatmap (V2)

**V1 alternative:** The Cost & Usage view includes a "Top 10 most-read files" and "Top 10 most-edited files" table, derived from `read_file` and `edit_file` tool call frequency. This covers the core signal without the visualization investment.

**V2:** Full visual heatmap with repository tree view, failure correlation, churn rate, and context cost dimensions.

#### FR-4: Quality & Verification Metrics

Quality metrics fall into two tiers based on data availability:

**Tier 1 — Available from agent telemetry (V1):**
- Session success rate: sessions ending in `completed` vs. `error` vs. `cancelled` status
- Error rate by category (see Section 4.0.6): API, Tool, Permission, Runtime
- Retryable error recovery rate: sessions that hit a retryable error and still completed
- Iterations-to-completion distribution: histogram of `TurnSummary.iterations`
- Tool error rate: `ToolResult` with `is_error: true` / total tool results

**Tier 2 — Requires CI/CD and SCM integration (V2+):**
- Test pass rate (first-pass vs. after retries)
- Lint/type-check pass rate
- Code review outcomes (approved/changes-requested/rejected)
- Regression detection
- Auto-fix rate

**V1 displays Tier 1 only.** Tier 2 requires GitHub/GitLab and CI/CD integrations that are out of scope for V1.

**Trend view (V1):**
- Session success rate over time (weekly rolling)
- Error category trends

#### FR-5: Team & User Breakdown

Enables managers to understand per-team and per-user patterns. Depends on enrichment pipeline (Section 4.0.8).

**Must show:**
- Per-team summary: sessions, sessions completed, cost, completion rate
- Per-user summary: same metrics at individual level
- Adoption curve: new users over time, usage frequency distribution
- Usage patterns: peak hours, preferred models

**Privacy consideration:**
- Individual user data visible only to their direct manager and org admins
- Team-level data visible to all team members

#### FR-6: Optimization Advisor (V2)

The full Optimization Advisor — a multi-stage pipeline with detection rules, impact estimation, ranking, feedback loops, and 12 recommendation types — is deferred to V2. It is a product in its own right and should not be bundled into the V1 dashboard launch.

**V1 alternative — Static Insights panel:**

A simple, rule-based insights panel on the Org Overview page showing up to 3 cost outliers:

1. **Highest-cost team this period** — "Team X spent $Y this week, Z% above org average"
2. **Lowest cache hit rate** — "Team X has a N% cache hit rate vs. org average of M%"
3. **Most expensive session** — "Session S cost $X (P times the average) — view details"

These insights are computed from the same data already available in FR-2 and require no ML pipeline, no feedback loops, and no additional infrastructure.

#### FR-7: Filtering & Time Controls

Global controls that apply across all views.

**Must support:**
- Time range: preset (today, 7d, 30d, 90d) and custom date range
- Team filter: single or multi-select
- User filter: single or multi-select
- Model filter: by AI model/provider

**Deferred to V2:** Repository filter (requires enrichment pipeline to reliably extract repo from sessions), task type filter (no "task type" concept in agent data model).

### 5.2 Data & Metrics

#### FR-8: Metric Definitions

All metrics must have consistent, documented definitions. The "session" is the unit of work in V1 (not "task" — see Review Log #3).

| Metric | Definition | Source | Unit |
|---|---|---|---|
| Session | A single agent conversation from start to termination | `Session` struct | count |
| Turn | One user message + all resulting assistant/tool exchanges | `TurnSummary` | count |
| Iterations | LLM API round-trips within a single turn | `TurnSummary.iterations` | count |
| Completion rate | Sessions ending with `completed` status / total sessions | Agent status | percentage |
| Cost | `(input_tokens x input_price) + (output_tokens x output_price) + (cache_creation x create_price) + (cache_read x read_price)` per model tier | `TokenUsage` + `ModelPricing` | USD |
| Cost per session | Total cost / completed sessions for a given scope | Derived | USD |
| Cache hit rate | `cache_read_input_tokens / (cache_read_input_tokens + input_tokens)` | `TokenUsage` | percentage |
| Adoption rate | Users with >= 1 session in period / total org users | Session metadata | percentage |
| Token efficiency | `output_tokens / (input_tokens + cache_creation_input_tokens)` | `TokenUsage` | ratio |
| Tool call rate | `ToolUse` blocks per session | `ContentBlock::ToolUse` | count/session |
| Tool error rate | `ToolResult` with `is_error: true` / total tool results | `ContentBlock::ToolResult` | percentage |
| Permission denial rate | PERM_xxx errors / total tool calls | Error events | percentage |

**Removed from V1:** "Task" (no such concept in agent data model), "First-pass quality" (requires CI/CD integration), "Agent utilization" (requires infra metrics), "Context utilization" (requires model context limit mapping — see Review Log #12), "File heat score" (V2), "Agent health composite" (V2).

#### FR-9: Data Granularity & Retention

- **Granularity:** Metrics stored at per-message level (each `ConversationMessage` with `TokenUsage`), aggregated to session/hourly/daily for dashboards
- **Refresh:** Standard views (Costs, Teams, Settings) load data on page open and filter change — no polling. Overview supports opt-in "Go Live" mode via SSE (see Section 4.0.7).
- **Retention:** Raw session JSON retained 90 days, daily aggregates retained 1 year
- **V1 scale target:** Single org, up to 1,000 users, 100,000 sessions/month. Estimated storage: ~5-10 GB/month at this scale. Storage backend must be specified during technical design.

## 6. Non-Functional Requirements

### NFR-1: Performance
- Dashboard initial load: < 2 seconds
- Filter/drill-down response: < 500ms
- Data freshness: metrics lag < 60 seconds from session completion to dashboard availability
- V1 scale: single org, up to 1,000 users, 100,000 sessions/month

**V2 scale target:** 10,000 users, 1M sessions/month.

### NFR-2: Accessibility
- Keyboard-navigable
- Semantic HTML throughout
- Screen-reader compatible charts (data tables as fallback)

**V1.1:** Full WCAG 2.1 AA audit and remediation.

### NFR-3: Responsive Design
- Optimized for desktop (primary)
- Usable on tablet (secondary)
- Mobile: read-only summary view

### NFR-4: Security & Authorization
- Role-based access: org admin, team admin, team member
- Org admins see all data; team admins see their team; members see their own data + team aggregates
- All API endpoints authenticated; no data leakage across org boundaries
- Audit log for dashboard access

### NFR-5: Internationalization
- USD as default currency
- Date/time displayed in user's timezone
- UI text externalized for future localization

**V2:** Multi-currency support (org-configured currency).

## 8. Success Criteria

### 8.1 Dashboard Adoption

| Metric | Target | How to measure | Timeframe |
|---|---|---|---|
| Weekly active users (WAU) | > 30% of org admins + team leads | Unique users with >= 1 dashboard session/week | 60 days post-launch |
| Return rate | > 40% of first-week users return in week 2 | Cohort retention: week-1 users who log in during week 2 | Weeks 1-2 |
| Session depth | Avg >= 3 views per session | Page/tab navigation count per dashboard session | 60 days post-launch |
| Time-to-insight | Median < 30 seconds from login to first filter/drill-down action | Time between session start and first interactive event | 60 days post-launch |

### 8.2 Cost Visibility

| Metric | Target | How to measure | Timeframe |
|---|---|---|---|
| Cost allocation coverage | 100% of agent spend attributable to team + user | (Sessions with team + user tags / total sessions) x 100 | At launch |
| Export usage | >= 1 cost export per org per month | Count of CSV exports from Cost & Usage tab | 60 days post-launch |
| Budget alert activation | >= 50% of orgs configure at least one budget alert | Orgs with active budget threshold / total orgs | 60 days post-launch |

### 8.3 Agent Reliability Visibility (replaces "Quality Confidence" — scoped to Tier 1 metrics)

| Metric | Target | How to measure | Timeframe |
|---|---|---|---|
| Error trend awareness | > 70% of team leads can identify their top error category when asked | Quarterly survey | 90 days post-launch |
| Session success rate visibility | Dashboard shows session success rate on Org Overview with < 60s data lag | Automated test | At launch |

### 8.4 Performance & Reliability

| Metric | Target | How to measure | Timeframe |
|---|---|---|---|
| Dashboard load time (p95) | < 2 seconds | RUM (Real User Monitoring) | At launch |
| Filter/drill-down response (p95) | < 500ms | Time from interaction to render complete | At launch |
| Data freshness | Metrics lag < 60 seconds from session end to dashboard | Timestamp comparison | At launch |
| Uptime | 99.9% (< 43 min downtime/month) | External health check | Monthly |

### 8.5 North Star Metric

**Cost per completed session, tracked weekly.**

- **Formula:** Total agent spend (tokens, computed from `TokenUsage` x `ModelPricing`) / count of sessions with `completed` status
- **Target:** Downward trend over any rolling 30-day window, for orgs actively using the dashboard
- **Why this is the north star:** It combines cost and reliability (only completed sessions count) into a single number derived entirely from available telemetry. If this number trends down, the dashboard is delivering value.

## 9. Open Questions

1. **Billing model (BLOCKING):** Are agent costs billed per-token, per-session, per-seat, or usage-based? This determines whether the cost dashboard is essential infrastructure or a nice-to-have. Must resolve before finalizing V1 scope. If usage-based, cost dashboard is the killer feature. If seat-based, adoption metrics matter more.
2. **Data source:** Do we ingest telemetry from the agent runtime directly, or consume pre-aggregated data from an internal API?
3. **Enrichment pipeline implementation:** What is the source of truth for user-to-team mapping? LDAP/SSO provider? Internal org management API? This is a prerequisite for FR-5.
4. **Storage backend:** V1 needs to handle ~5-10 GB/month of session data with 90-day raw retention. What is the storage strategy? (Postgres + S3? Clickhouse? Must be decided in technical design.)

## Stage 1 Additions (2026-04-05)

### FR-10: Real Data Pipeline

**Status:** Implemented

The dashboard now connects to real databases instead of in-memory mock data.

| Component | Implementation |
|---|---|
| **Event ingestion** | Rust simulator generates streaming-level agent events (SessionStart, TextDelta, ToolUse, ToolResult, Usage, MessageStop, SessionEnd) matching the Claw Code reference agent format |
| **Raw storage** | TimescaleDB hypertable — one row per streaming event, partitioned by timestamp |
| **Message queue** | Kafka (optional) — decouples simulator from transformer, enables replay |
| **Transformation** | Rust transformer aggregates events into per-endpoint PostgreSQL tables |
| **Dashboard storage** | PostgreSQL — pre-aggregated baked tables, one per API endpoint |
| **API** | TypeScript backend reads PostgreSQL directly with type-safe queries (pg + zapatos) |

### FR-11: Live Dashboard via SSE

**Status:** Implemented

| Feature | Implementation |
|---|---|
| **Real-time overview** | SSE polls PostgreSQL every 5s, pushes KPIs/trends to frontend |
| **Budget SSE** | Separate SSE channel for budget state — broadcasts on every save |
| **Turbo mode** | 15 updates/sec with no DB — for UI stress testing |
| **Auto-connect** | Overview connects to live SSE on page load, disconnects on navigation |

### FR-12: Server-Side Budget Management

**Status:** Implemented

Budget logic moved from client to server:

| Feature | Implementation |
|---|---|
| **Auto-expansion** | When a team override exceeds its current allocation, org budget expands by the delta |
| **Validation** | Rejects saving org budget below override sum (hard error, not warning) |
| **Per-team endpoint** | `POST /api/budgets/team` — send `{ teamId, budget }`, server merges and expands |
| **SSE broadcast** | Every budget save broadcasts updated state to all connected clients |
| **Auto-distribution** | Remaining budget / non-overridden teams. $0 when budget fully consumed by overrides |

| Feature | Implementation |
|---|---|
| **Devbox** | Reproducible environment: Node 20, pnpm 9, Rust, cmake, librdkafka |
| **Two terminals** | `cargo run -- poll` + `pnpm dev` to run the full stack |
| **Poll mode** | Seeds 3 days + generates ~900 sessions every 30s, writes directly to both DBs (no Kafka needed) |
| **Observation tools** | `status`, `verify`, `kafka-monitor` subcommands for inspecting pipeline state |
| **Documentation** | ENTRY_POINT.md navigation guide, TECH_SETUP.md operational reference |
